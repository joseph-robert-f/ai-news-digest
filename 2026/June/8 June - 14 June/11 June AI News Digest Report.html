Top Story — Developer Agents Cross the Default-On Line

MicrosoftVS Code 1.124 ships Copilot Autopilot on by default, with a utility model deciding when a task is “done”

10 June 2026 · Primary source: code.visualstudio.com release notes / Microsoft Build engineering blog

Microsoft released Visual Studio Code 1.124 on Wednesday 10 June. The headline change is that Copilot Autopilot — the chat permission level that lets agents take initiative and act without per-step user approval — is now enabled by default for all users, having shipped behind a feature flag in 1.123. The agent loop itself was rebuilt: rather than the previous fixed “finish-when-no-more-edits” rule, a small utility model now reads the chat transcript and judges whether the task is genuinely complete, with a hard cap of three iterations in the Advanced Autopilot mode before the loop terminates and returns control to the developer. The release also adds the new Agents window, a dedicated companion surface for exploring, iterating on, and reviewing agent sessions across projects and machines; background sessions so a developer can dispatch a request and immediately compose the next; keyboard-driven session navigation (search, jump, step); and a searchable browser history inside the integrated browser. The practical implication for engineering orgs: every VS Code installation that upgrades this week now ships with a multi-step autonomous coding agent enabled out-of-the-box, with vendor-side guardrails replacing user-side micromanagement as the default trust model. Coverage in Neowin, TechTimes, and Windows Report converges on the same characterization — the autopilot-by-default flip, not any individual feature, is the news.

Software Engineering & Supply-Chain Security

npmMiasma worm second-wave fallout: “Phantom Gyp” abuses binding.gyp to bypass install-script monitoring

Disclosed 1–3 June, with ongoing CI/CD fallout flagged this week · Primary source: Microsoft Security Blog / StepSecurity

The Miasma npm worm — a self-propagating credential-harvesting campaign built on the publicly released Mini Shai-Hulud codebase — remains the open supply-chain story this week as enterprises continue to audit blast radius. The initial wave on 1 June compromised 32 @redhat-cloud-services packages (~80,000–117,000 weekly downloads in scope) via unauthorized commits pushed to repositories in the RedHatInsights GitHub organization; the attack was traced to a single compromised employee GitHub account whose credentials had appeared in commercial infostealer logs as early as 13 April — nearly seven weeks of dwell time before weaponization. On developer systems the malware exfiltrated SSH keys, CLI credentials, and browser / wallet data; in CI/CD it scraped GitHub Actions runner memory for secrets, escalated via passwordless sudo, and republished poisoned packages with forged SLSA provenance. The 3 June second wave introduced “Phantom Gyp”: a 157-byte binding.gyp file that triggers code execution during npm install — not via the preinstall or postinstall lifecycle scripts that most install-time security tooling monitors. 57 packages across 286+ malicious versions were published in a rolling campaign lasting under two hours. The operational takeaway flagged by Microsoft Security, StepSecurity, Snyk, and Orca: install-script allow/deny lists are now structurally insufficient; build-graph monitoring and short-lived CI credentials are the minimum bar. Any environment that ran npm install against the affected scopes in the disclosure window should treat all reachable secrets as exposed.
